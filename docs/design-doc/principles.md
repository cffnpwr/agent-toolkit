# 設計原則

agent-toolkit全体に共通する設計上の原則を定める。
個々の設計判断はこれらの原則に従う。
この原則は現行の成果物だけでなく、本リポジトリに今後追加するすべてのスキル・hookに適用する。

## ホスト充足を前提とする

APM（Agent Package Manager）は各primitive（スキル・hook）をターゲットへコピーするが、primitiveが実行時に必要とする言語パッケージ・システムバイナリは導入しない（[全体アーキテクチャ](./architecture.md)を参照）。
各成果物は自身の外部依存を明示宣言し、ホスト環境で充足されることを前提に動作する。
依存の充足は配布・環境側の責務であり、成果物の責務ではない。

## 依存を明示宣言し再現性を確保する

外部依存はすべて明示的に宣言する。
依存パッケージ（言語のライブラリ）は、同梱した定義ファイルとロックファイルでバージョンを固定し、依存解決を再現可能にする。
外部ツール（汎用CLI・インタプリタ）は、バージョン要件と存在確認とともに列挙する。
宣言の詳細は[外部ランタイム依存](./dependencies.md)で定める。

## 再現的な操作のみ自律実行する

リポジトリ内でgitが検知するファイル更新を生まずに実行できる再現的操作は、停止・エスカレーションなしに実行してよい（対話時はAI Agentが、非対話のhookは起動スクリプトが行う）。
コミット済みロックファイルからの依存同期（`uv sync --frozen`等）は、`.venv`・`node_modules`などgit管理外のみを変更するためこれに該当する。
依存の追加・更新、外部ツールの導入、代替手段へのフォールバックは行わず、進行を停止してユーザーへエスカレーションする（非対話のhookは停止の代わりにfail-openで警告して通す。[Hook機構](./hooks.md)）。

## Harness非依存に設計する

成果物は複数のHarnessへ配布される。
ここでのHarnessは、スキル・hookのプロトコルを実装するホスト実行環境（Claude Code、Codex CLI、Gemini CLI、GitHub Copilot CLI等）を指し、推論を行うAIモデルとは区別する。
特定Harness専用の制御手段に依存せず、対応する全Harnessで成立する手段へ一本化する。
対応できないHarnessでは無作用へ劣化させ、誤判定で操作を阻害しない。
