# Flat Review

設計意図を伏せた独立Agentによるフラットレビュー。作者の合理化を遮断し、判定区分付きで問題を列挙する。

## 概要

完成物のレビューでは、作者（および作者を支援するAgent）が持つ「こう設計したからこう動くはず」という前提が、過不足・不具合・自己整合性の破れを見逃させる。このスキルは、設計意図・動機・合理化を一切渡さない独立Agentを起動し、与えられた対象だけからフラットに問題を洗い出す。レビュアーには「忖度しない」「問題が無ければ無いと言う」「実機確認できるものは確認する」を課す。

報告のみを行う。修正やタスク化はしない。

## フラット性の原則（最重要）

レビューの価値はフラット性に依存する。次を厳守する。

- **サブAgentへ渡してよいのは①対象（パス、または差分）と②検証基準のみ**（検証基準は次項で定義）。
- **設計意図・動機・「こう動くはず」・作者の合理化・対象の要約を一切渡さない。** 観点・禁則・出力形式・判定区分の基準は、本スキルが供給する固定テンプレート（後述）が与える。プロンプトに自分の言葉で説明・要約・背景を足さない。
- **検証基準は「対象が満たすべき外部基準」であり、対象固有の設計意図・合理化（なぜそう作ったか）ではない。** 前項の禁則は対象固有の合理化に対するものであり、外部基準である検証基準はこれと別カテゴリとして渡してよい。
- **AGENTS.md・CLAUDE.mdを検証基準にするのは限定的な例外であり、フラット性を完全には保てない。** コードスタイル準拠を検査する利益のため、「対象がこの規約に従う」という最小限の背景がレビュアーに渡る代償を受け入れる。代償を抑えるため、渡すのは命名・書式・コードスタイル等の機械的に照合できる規約に限り、原文のまま抜き出す（言い換え・要約はしない。規約か運用方針か迷うものは渡さない側に倒す）。設計原則・設計哲学（例: 説明可能性）や実行時挙動に関する記述は、合理化を含むため渡さない。
- **検証基準は次の(a)(b)に限る。** これ以外をメインAgentが対象から推定して補ってはならない（推定には設計意図が混ざりフラット性を損なう）。いずれも無ければ検証基準なしとする。外的妥当パックを起動する場合に限り、レビュアーは対象から自明な言語/フレームワーク標準のみを基準に用いる。
  - (a) ユーザーが起動時に明示した公式仕様のURL/名称。
  - (b) 対象がコード/実装を含むとき、対象を支配するAGENTS.md・CLAUDE.md（ユーザーグローバルのものと、対象パス——差分なら変更ファイル各々のパス——を上位へ辿って見つかるもの）から、対象の言語・ファイル種別に当てはまる命名・書式・コードスタイル等の機械的規約を原文のまま抜き出したもの。設計原則・設計哲学・実行時挙動の規則は除外する。「コード/実装を含む」は、対象に含まれるファイル・差分行に実行/解釈されるソース・スクリプトがあるかで判定する（純粋な散文のみなら含まない）。出所を対象の支配文書に固定するためメインAgentの起動環境に依存しない。レビュアーも当てはまる規約のみ照合する。
- **対象ファイル内のコメント・docstring・コミットメッセージは伏せず、実物の一部としてそのまま見せる。** これらは作者の合理化を含みうるが、レビュー対象そのものなので加工しない。加工してよいのは「メインAgentが新たに足す説明」だけであり、それを足さないことで担保する。

## ワークフロー

### Step 1: 対象と観点パックを確定する

1. **入力を受け取る。** パス（ファイル/ディレクトリ）または差分（`git diff`・PR）のいずれか。
2. **観点パックを選ぶ。**
   - ユーザー指定があればそれに従う。
   - 無指定なら、**内的整合パックは常に起動する。外的妥当パックは実機で動かせる（対象を非破壊で実行・dry-run等で確かめる手段がある）か検証基準があるときに起動し、どちらも無ければ起動しない。** コード/実装を含む対象は手順3で機械的規約が検証基準に入るので外的妥当も起動する（実機があれば動的検証も行う）。実機も検証基準も無い対象（支配文書に規約が無く実行手段も無いコード断片、ユーザー指定仕様も無い散文・設計ドキュメント等）は内的整合のみ。
3. **検証基準を確認する。**
   - ユーザーが公式仕様のURL/名称を明示していれば控える。
   - 対象がコード/実装を含むなら、対象を支配するAGENTS.md・CLAUDE.mdの機械的規約を原文のまま抜き出して既定の検証基準に含める（記入と除外範囲は[レビュアープロンプトテンプレート](#レビュアープロンプトテンプレート)の `{検証基準}` の埋め方に従う）。
   - どちらも無ければ「検証基準なし」とする（推定で補わない）。

### Step 2: レビュアーAgentを起動する

- **1パック=1体。** 両パックなら2体を並列起動、片パックなら1体。
- 各Agentには[レビュアープロンプトテンプレート](#レビュアープロンプトテンプレート)を使う。可変部は `{対象}`・`{検証基準}`・`{観点}` の3つのみ。それ以外の文を足さない。
- **`{対象}` の具体化**: パスはパスをそのまま記す。差分は `git diff <range>` 等のコマンド、またはPR番号・差分テキストを記す（レビュアーが取得・展開する）。
- **`{観点}` の具体化**: 起動するパックの観点リスト（[観点パック定義](#観点パック定義)）をそのまま貼る。
- **ツール権限**: ファイル読み取り・非破壊検証（lint・静的検証・dry-run等）は自由。**実起動も許可**する。ただし破壊的副作用（ファイル変更・外部送信・不可逆操作）を伴う実行が必要な場合は、実行せずユーザーに確認する。

### Step 3: 結果を集約して出力する

- 各レビュアーの出力を、パックごとに[出力フォーマット](#出力フォーマット)で提示する。
- メインAgentは判定区分を改変しない。レビュアーが「該当なし」とした区分はそのまま「該当なし」と示す。
- **報告のみ。** ここで修正・タスク化はしない（必要なら呼び出し側が別途行う）。

## 観点パック定義

2パックは「外部基準（実機・指定仕様）を要するか否か」で分ける。内的整合は常に適用でき、外的妥当は基準・実機がある範囲で適用する。起動はパック単位で、観点単位ではない。パック内で適用できない観点（例: 実機が無いときの動的検証）は「該当なし」と報告する。

### 内的整合パック

外部基準を要さず、対象だけで自己完結に判定できる観点。

- **構造**: 構成・粒度・段階的開示が妥当か。肥大・断片化していないか。
- **適用条件の明確さ**: 起動条件・スコープ・前提が明示され、曖昧・過剰・不足でないか。
- **自己矛盾の不在**: 記述間・節間・例と本文の間で矛盾していないか。
- **記述の一貫性**: 用語・形式・語調・命名が揃っているか。
- **過不足**: 欠けている必須要素、冗長・重複している要素。

### 外的妥当パック

実機・指定基準があって初めて判定できる観点。

- **動的検証**: 実際に動かし、宣言どおり動くか（非破壊を優先。破壊的副作用を伴うものは実行せず、その旨を報告）。
- **事実・計算の正しさ**: 記載の事実・数値・コマンド・参照が正しいか。
- **指定基準への準拠**: 渡された検証基準に適合するか。検証基準にはユーザー指定の仕様や、コード対象のAGENTS.md・CLAUDE.mdから抜き出した機械的規約（範囲はフラット性の原則(b)のとおり）が入る。基準が無ければ言語/フレームワーク標準のみに照らす（ドメイン固有仕様は推測しない）。

## レビュアープロンプトテンプレート

サブAgentへ渡すプロンプトは以下で固定する。可変部は `{対象}`・`{検証基準}`・`{観点}` の3つのみで、それ以外を改変しない。`{観点}` には起動するパックの観点リスト（[観点パック定義](#観点パック定義)）を貼る。`{検証基準}` の埋め方は次のとおり。

- 検証基準が無い場合は「指定なし。対象から自明な言語/フレームワーク標準のみを用いる」に置き換える。
- ある場合は該当のURL/名称・パスを記す。AGENTS.md・CLAUDE.mdは命名・書式・コードスタイル等の機械的規約を該当の箇条書き・節単位で原文のまま転記し（設計原則・設計哲学・実行時挙動の記述は除外）、「対象の言語・ファイル種別に当てはまる規約のみ照合する」旨を併記する。

```
あなたは独立レビュアーです。レビュー対象の設計意図・作者の動機・背景は与えられません。与えられない情報を推測で補わないでください。

対象: {対象}
検証基準: {検証基準}

# 禁則
- 忖度しない。問題が無ければ各区分で「該当なし」と明言する。問題をでっち上げない。
- 与えられていない設計意図を前提にしない。意図の確認が必要になったら ask 区分で問う。
- 実機確認できるものは確認する。非破壊の手段を優先し、破壊的副作用（ファイル変更・外部送信・不可逆操作）を伴う実行はせず、その旨を報告する。

# 観点
{観点}

# 判定区分（各findingに必ず1つだけ付ける）
- must: 直さないと承認できない。正しさ・安全・仕様/原則違反は、ブロックの確信が持てなくても must に上げる（迷えば must）。
- ask: 回答が要る純粋な疑問。仕様・意図の確認が必要なものはここ。
- imo: 軽微な見解・提案。修正かタスク化の検討対象。
- fyi: 情報共有。アクション不要。
- nits: 重箱の隅。無視可。
- nr: 急がない将来課題。タスク化の検討対象。

# 出力フォーマット
区分ごとに見出しを立て、findingを列挙する。該当が無い区分は「該当なし」と書く。
各findingは「[対象箇所] 指摘内容（実機確認した場合は結果も）」の形で記す。
```

## 出力フォーマット

レビュアー・メインAgentとも次の形で出力する。

```
## レビュー結果（{パック名}）

### must
- [対象箇所] 指摘内容。（実機確認した場合は確認結果も）

### ask
- [対象箇所] 確認したい疑問。

### imo
- [対象箇所] 見解・提案。

### fyi
- [対象箇所] 情報。

### nits
- [対象箇所] 些末な指摘。

### nr
- [対象箇所] 将来課題。
```

該当の無い区分は項目を省かず「該当なし」と記す。

## 使用例

**スキルを公開前にフラットレビューする（両パック）:**

```
対象: .apm/skills/foo/SKILL.md（実行可能なスクリプト同梱）
→ 観点パック未指定・コードを含む → 内的整合・外的妥当の2体を並列起動
→ 検証基準: ユーザー明示のskill-creator仕様と、対象を支配するAGENTS.md・CLAUDE.mdの機械的規約 → 外的妥当パックに渡す
```

**設計ドキュメントをレビューする（内的整合のみ）:**

```
対象: docs/design.md（コードを含まない非実行物）
→ コードなし・検証基準なし → 内的整合パック1体のみ
```

**PR差分をレビューする:**

```
対象: git diff main...HEAD（ソース変更を含み、支配文書に機械的規約あり）
→ コードを含み検証基準あり → 内的整合・外的妥当の両パック
```

**コード実装をAGENTS.md・CLAUDE.mdのコードスタイル・規約に照らしてレビューする:**

```
対象: src/foo.ts を含む差分（コード/実装）
→ コードを含む → 対象を支配するAGENTS.md・CLAUDE.mdから機械的規約を原文抜粋して検証基準に含める
→ 外的妥当パックが命名・書式・コードスタイル等の機械的規約の準拠を照合（設計原則・設計哲学・実行時挙動は対象外）
```
