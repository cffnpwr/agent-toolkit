# Test Writing

テストの書き方を、言語非依存の方針と言語固有の手順に分けて示すガイド。

## 概要

テストの粒度・何をテストするか・命名・構造・モックの使いどころは言語を横断して共有できる。このスキルはその核を本文に置き、フレームワーク固有の具体手順（ランナー・配置・パラメタライズ・カバレッジ・境界モック）を `references/<言語>.md` に分ける。テストの新規執筆と既存テストのレビュー/改善の両方に適用する。

## 最初にすること

1. 対象コードと依頼から言語を判定する。
2. 該当する reference を読む。
   - Rust → [references/rust.md](references/rust.md)
   - Go → [references/go.md](references/go.md)
   - TypeScript → [references/typescript.md](references/typescript.md)
   - 一覧に無い言語 → reference は無い。本文の言語非依存方針のみを適用し、命名・配置・カバレッジ計測などの具体手順は対象言語の標準テストランナーの公式ドキュメントとプロジェクトの既存慣行に従う。
3. 本文の言語非依存方針と、reference の言語固有手順を合わせて適用する。

言語固有の道具（アサーション・モック・カバレッジ計測）は、対象プロジェクトの既存慣行があればそれを優先し、無ければ reference の定番に従う。

## テスト方針（言語非依存の核）

### 粒度

テストピラミッドに従う。高速・安定な unit テストを土台に多数置き、integration は要所に絞り、e2e はごく少数にする。

### 何をテストし、何をしないか

- **public な呼び出し経路は C1（分岐網羅）を満たすようにテストを書く。** 各条件分岐の真・偽の双方を通すケースを用意する。
- **C1 の対象は到達可能な分岐に限る。** 到達不能・防御的な分岐（`unreachable!`、網羅 `match` の default 腕、実質起こり得ないエラー側など）は対象外とする。それらを通すためだけに注入点やモックを増設しない（後述「モック / テストダブル」の境界のみダブル化を優先）。可能なら型で分岐自体を消す。
- **private でも、ロジックを持つ内部要素はテストする。** 単なる委譲・受け渡しでなく判断や計算を含むものが対象。
- **単純なコードはテストしない。** 自明な getter・素通しのラッパ等はケースを起こさない。
- **IO のようにテストが難しいものは無理にテストしない。** 外部境界そのものの検証にコストをかけず、境界を跨ぐロジック側をテスト可能な形に寄せる。

### 命名

検証する条件と期待結果が名前から読めることを原則とする。正常系・異常系を prefix で区別する。

- **テストケースごとに関数を書く言語（例: Rust）**: `positive` / `negative` を prefix に付け、以降は言語の命名慣習に従う。
- **テストケースを文字列等で自由記述できる言語（例: Go, TypeScript）**: `[positive] ` / `[negative] ` を prefix に付け、以降は「A が B のとき C（となる）」のように入力条件と期待結果がわかるよう、文の主題を明確にして書く。

### 構造

Given-When-Then を基本パターンとする（準備 → 実行 → 検証）。

- **BDD スタイル（describe/it 等）の場合**: テスト対象ごとに `describe` を分ける。
- **テーブルドリブン（TDT）の場合**: テスト対象ごとにテスト関数を分ける。

### 検証（アサーション）

- **正常系は期待値全体との完全一致で検証する。** 期待値のオブジェクト / 構造体を 1 つ組み立て、結果全体を一度に突き合わせる。フィールドごとの部分検証はしない（アサーション行が増え、検査対象から漏れたフィールドの差分に気づけないため）。
- **異常系はエラーの型・種別まで検証する。** 期待するエラー型 / バリアントに一致することを検証する。**エラーメッセージの文言は検証しない**（実装変更で容易に壊れ、検証の意図も型ほど明確でないため）。

### モック / テストダブル

外部境界（ネットワーク・DB・時刻・乱数など）のみ必要最小限でダブルに置き換える。内部モジュール同士はモックせず実物で繋ぐ。境界を差し替えたくてもできない場合は、依存を境界へ寄せる設計を検討する。

## カバレッジ計測の言語差（注意）

C1（分岐網羅）を機械計測できるかは言語・ランナーで異なる。計測できない場合は、各分岐に対応するケースを人手で用意して C1 を担保する。詳細は各 reference を参照する。

| 言語 / ランナー | C1（分岐）の機械計測 |
| --- | --- |
| Rust (`cargo-llvm-cov`) | region coverage が既定。真の分岐カバレッジは nightly + `--branch`（unstable） |
| Go (`go test -cover`) | 不可（文カバレッジのみ）。ケースで人手担保 |
| TypeScript (`vitest`) | 可（`--coverage`） |
| TypeScript (`bun test`) | 不可（分岐未対応）。ケースで人手担保 |
