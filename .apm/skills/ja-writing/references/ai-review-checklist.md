# AIによる補完レビューのチェックリスト

textlintのカバレッジ外のため、手動またはAIによる確認が必要な項目。使用方法は[../SKILL.md](../SKILL.md)の「AIによる補完レビューの手順」を参照する。

## 語尾の長音符号

`@textlint-ja/no-synonyms`はSudachiの類義語辞書を使い表記ゆれ（例: `サーバー / サーバ`）を検出するが、辞書は全ての外来語をカバーしない。

外来語の語尾長音符号（ー）は常に付ける（文化庁「[外来語の表記](https://www.bunka.go.jp/kokugo_nihongo/sisaku/joho/joho/kijun/naikaku/gairai/index.html)」に従う）。

| 誤り         | 正しい         |
| ------------ | -------------- |
| サーバ       | サーバー       |
| コンピュータ | コンピューター |
| ユーザ       | ユーザー       |
| フォルダ     | フォルダー     |
| プリンタ     | プリンター     |

`@textlint-ja/no-synonyms`を一次チェックとして使い、残りの外来語は手動で確認する。

## 文体統一の抜け漏れ

`.textlintrc.json`の`no-mix-dearu-desumasu`は本文中の常体・敬体混在を検出するが、次は検出しない。

- コードコメント内の文体混在
- 引用（blockquote）内の文体混在
- 本文を持たない文書（純粋なリストのみ等）での文体混在

常体・敬体はどちらでもよいが、文書内で統一する。

- 常体: `設定する`、`変更できる`、`動作する`
- 敬体: `設定します`、`変更できます`、`動作します`

READMEやユーザー向けドキュメントは敬体、技術仕様書や内部ドキュメントは常体が典型的な慣例。

## 文章構成・論証の妥当性

構造・論理に関わる次の項目はtextlintで機械的に検出できない。

- 段落構成: 一段落一トピックか、段落冒頭が前段落との論理関係を示しているか
- 論証の妥当性: 推量と断定の使い分けが妥当か、複数要因を単一原因に還元していないか、因果の機構を説明しているか、主張と例が対応しているか
- 読み手負荷: 固有名詞・新規文脈の追加が必要か、省略すべき詳細を残していないか
- 視点・語り: 行為者を主語にできる箇所を受身のままにしていないか、術語が一貫しているか、役割名と「あなた」の使い分けが妥当か

## 表現・演出

頻度や文脈への依存が強く、textlintの単純なパターン検出では正当な用法と区別できない項目。

- 「さらに」「また」「加えて」を同一段落内で連続使用していないか
- 「まとめると」「要するに」が直前の内容の言い換えに留まっていないか（内容を圧縮した真の要約であれば許容）
- 「鍵となる」「根本的な」「包括的」「総合的」「非常に」「極めて」「大いに」が、具体的な根拠を伴わない強調になっていないか
- 「正面から扱う／回収する／見る／書く／立てる」のように、内容を示さず姿勢だけを宣言していないか
- 「〜と言えるだろう」が、推量・仮定・読者の疑念を表すのでなく根拠なく主張を弱めていないか
- 「AではなくBだった」という対句を多用していないか
- 「必ず」を、検出・保証・解決の文脈で無条件の確実性を装うために使っていないか

## 整形

- ダッシュ（em `—`・horizontal bar `―`・二倍ダッシュ`——`）を地の文・見出しで使っていないか（範囲を示すenダッシュ`–`・英語の複合語・コードブロック・書誌情報は対象外）
- 一文ごとに改行し、段落を空行で区切っているか
- コード・ログ・設定ファイルの断片をコードブロックで示しているか、本筋から外れる補足を脚注に降ろしているか
