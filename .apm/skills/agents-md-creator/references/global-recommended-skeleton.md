# 推奨骨格の定義（グローバルモード）

グローバル（ユーザーレベル・全プロジェクト共通）のAGENTS.md本体に置く推奨セクション群と、各セクションの採否判定基準。**該当しないセクションは出さない**（空セクションや「なし」を残さない）。順序は推奨であり、ユーザーの関心に応じて調整してよい。

グローバル設定は大半が**コードから読めないユーザーの選好（意図）**なので、各セクションの内容は面談で収集する（[global-content-gathering.md](global-content-gathering.md) 参照）。リポジトリ用骨格（プロジェクト概要・ビルド/テスト/lintコマンド・ディレクトリ構成）は**グローバルでは使わない**。特定のプロジェクトに依存しない、全プロジェクト横断で常時適用される選好だけを書く。

本体に置くのは「どのプロジェクト・どのタスクでも常時適用される情報」だけ。詳細手順・具体例・状況依存ガイドは設定ディレクトリ直下の `docs/` に切り出す（[progressive-disclosure.md](progressive-disclosure.md) 参照）。

## セクション一覧と採否基準

### 1. コミュニケーション・言語

- **内容**: エージェントが応答に使う言語、文体（敬体/常態など）、トーン（簡潔さ、客観性、クッション表現の可否）。**「何語で・どんな態度で」話すか**に限る。出力の構造・報告の粒度は「応答スタイル」へ分ける。
- **採否**: 言語・トーンの選好がある場合に入れる。多くのユーザーで該当する。
- **収集元**: 面談。既存グローバル設定があれば取り込む。

### 2. 応答スタイル

- **内容**: 出力の**構造・形式**。結論先出しなどの構成方針、完了報告の粒度、出力フォーマット、禁止する言い回し（締めの挨拶・前置き等）。
- **採否**: 応答の形に対する選好がある場合に入れる。
- **収集元**: 面談。
- **「コミュニケーション・言語」との境界**: 1は「何語で・どんなトーンか」、本セクションは「どう構造化し、どこまで報告するか」。言語・トーンは1へ、構造・報告・フォーマットは本セクションへ振り分け、重複セクションを作らない。

### 3. 作業の進め方・意思決定

- **内容**: タスクへの取り組み方と、確認するか進めるかの判断基準。依頼スコープの厳守（依頼されたことだけを行う）、要件は確認するが実装手段は自分で決める切り分け、要件・仕様の推測禁止、計画承認後の再確認回避など。
- **採否**: 作業の進め方に関する方針がある場合に入れる。実グローバル設定で比重が大きい中核セクション。
- **収集元**: 面談。コードからは読めない意図の中心。推測でデフォルトを捏造しない。
- **「横断的禁止事項」との境界**: 本セクションは「どう進めるか」の原則。単発の禁止行為（触れてはいけない領域等）は「横断的禁止事項」へ。

### 4. 自律性・権限

- **内容**: 確認なしで実行してよい操作の範囲、確認が必要な操作、破壊的操作の扱い。VCS操作の自律性（フィーチャーブランチでの操作許可、許可される破壊的VCS操作）もここに含める。
- **採否**: 「どこまで勝手にやってよいか」の方針がある場合に入れる。
- **収集元**: 面談。

### 5. ツール・環境の使用方針

- **内容**: 優先して使うツール、ツール選択の規則、インストール方針（グローバルインストールの可否、パッケージマネージャ・宣言的管理の制約）、調査・検索手段の優先順位。**既定のVCS（git/jj等）**もツール選択としてここに含める。
- **採否**: ツール選好や環境固有の制約がある場合に入れる。
- **収集元**: 面談。環境事実（OS・シェル・パッケージマネージャ）は補助的に読んでよいが、方針はユーザー確認。

### 6. 横断的禁止事項・制約

- **内容**: 全プロジェクトで守る単発の禁止事項・制約（例: 特定操作の禁止、触れてはいけない領域）。
- **採否**: 横断的な制約が存在する場合に入れる。コードから読めない意図なので面談で確認。
- **収集元**: 面談。
- **「作業の進め方」との切り分け**: 進め方の原則は3へ、個別の禁止行為は本セクションへ。

### 7. 横断的なコードスタイル・方針（任意）

- **内容**: 言語・フレームワーク非依存で全プロジェクトに成り立つコードスタイル・設計の選好（例: 厳格な型付け、関数型志向、DRY/KISS/YAGNI、明示的なエラーハンドリング）。
- **採否**: 全プロジェクト横断で適用する普遍的な方針がある場合に入れる。複数ツール・実践記事の横断調査でグローバル設定の頻出カテゴリ。
- **収集元**: 面談。
- **制約**: **言語/プロジェクト固有のコーディング規約・命名規則・アーキテクチャ境界は書かない**（リポジトリ側AGENTS.mdの領分。グローバルに置くとプロジェクト内容の混入を招く）。下記「グローバル/プロジェクトの境界基準」の普遍性・永続性テストを満たすものだけを残す。

### 8. その他の状況依存セクション（任意）

ユーザーの関心に応じて以下を検討する。該当しなければ出さない。

- **VCS運用**: 既定VCSの自律性・選択は4・5に主分散する。ブランチ戦略・コミット/PRの細かい運用などこだわりがある場合のみ、独立セクションとして立てる。
- **成果物の扱い**: コメント・注釈の残し方、変更履歴を成果物に書かない方針など。
- **セキュリティ・安全性の一般方針**。

## グローバル/プロジェクトの境界基準

ある項目をグローバルに書くか迷ったら、次のテストで判定する。

- **「自分について」か「プロジェクトについて」か**: 自分の作業の好み・横断的な選好 → グローバル。特定リポジトリの事実・チーム共有の規約 → プロジェクト側AGENTS.md。
- **普遍性・永続性テスト**: 「これは全プロジェクトで成り立ち、長期的に変わらないか?」 Yes ならグローバル、特定プロジェクト/特定時期の詳細ならプロジェクト側。

## アンチパターン（グローバルに入れない）

- プロジェクト固有のビルド/テストコマンド、命名規則、ディレクトリ構成、アーキテクチャ境界。
- グローバルとローカルの混在（"sludge"）。半分が汎用すぎ半分が固有すぎる状態は、全タスクに無関係な情報を運ばせる。
- マニフェスト・設定・言語標準から推論できる情報の重複記載（自明な指示も含む）。
- 古い/非推奨パターンの残存。削除されないまま積み上がると、誤った指示として害になる。

## 適応の原則

- 推奨骨格は出発点。ユーザーに該当しないセクションは**作らない**。
- グローバルは特定リポジトリに紐づかない。プロジェクト固有の事実（ビルドコマンド・ディレクトリ構成・コーディング規約等）は書かず、横断的な選好だけを残す。
- 1つのセクションに常時適用情報と状況依存情報が混在する場合、常時適用分だけを本体に残し、残りを `docs/` へプレーン参照で誘導する。
- セクション名・粒度はユーザーの使用言語と慣習に合わせてよい。
