# ルーブリック監査チェックリスト（グローバルモード）

既存のグローバルAGENTS.md（または各ツールのネイティブ設定）を改善する際の監査観点。各観点を評価し、問題があれば**差分提案**としてユーザーに提示する。修正のうちコードから読めない意図に関わる部分はユーザーに確認する。

グローバルは特定リポジトリに紐づかないため、リポジトリ実態とのドリフト監査は行わない。代わりに環境・ツールとの整合性、symlink構成、内部の一貫性を見る。

## 観点1: 推奨骨格の網羅

- [global-recommended-skeleton.md](global-recommended-skeleton.md) の推奨セクションのうち、**該当するのに欠けているもの**がないか。
- 逆に、**該当しないのに残っているセクション**（空セクション、「なし」だけのセクション）がないか → 削除を提案。
- リポジトリ固有の事実（特定プロジェクトのビルドコマンド・ディレクトリ構成等）が紛れ込んでいないか → グローバルから除去を提案。

## 観点2: 段階的開示の遵守

- 本体に**特定タスク時のみ必要な詳細**（込み入った手順、具体例、状況依存ガイド）が混入していないか → `docs/` への切り出しを提案。
- `docs/` に**常時適用すべき原則・選好**が埋もれていないか → 本体への昇格を提案。
- 判定は [progressive-disclosure.md](progressive-disclosure.md) の「常に読むべきか」軸で行う。

## 観点3: 環境・ツールとの整合性

- 記載されたツール使用方針が現在の環境と食い違っていないか（例: 存在しないパッケージマネージャ・廃止したツールへの言及）。
- もう使っていないツールやワークフローへの言及が残っていないか。
- 環境事実（OS・シェル・パッケージマネージャ）は [global-content-gathering.md](global-content-gathering.md) のステップ1で読み取り、既存記述と突き合わせる。

## 観点4: プレーン参照の参照切れ

- 本体から `docs/*.md` への参照先ファイルが**実在するか**。
- 参照切れがあれば、参照の削除か対象ファイルの作成を提案。
- 機械的にチェックしたい場合は [check_references.sh](../scripts/check_references.sh) を使ってよい（実体のAGENTS.mdのパスを渡す）。

## 観点5: symlink構成の正否

- ネイティブ設定（`~/.claude/CLAUDE.md` 等）が実体のAGENTS.mdへの正しい symlink になっているか、または実体ファイルとして内容が二重管理されていないか。
- 判定基準と提案手順は [global-entrypoint.md](global-entrypoint.md) を参照（提案は保守ポリシーに従い実行エージェントのツールのみ）。
- 実体のAGENTS.md本体に `@`参照がある場合、それが「常時コンテキストに読み込ませたい」という設計意図に沿った意図的なものか確認する。意図的なら許容、必要時のみ読めばよい情報なら プレーン参照への変更を提案、意図しない混入なら除去を提案。
- グローバルは複数ツールへ symlink で共有されるため、`@`参照は**自動展開に対応しないツール（symlink先の非Claude等）で黙って失効する**点に注意する。ツール横断で常時適用したい内容を `@`参照に頼らせていないか確認する。

## 観点6: 本体の肥大化

- 本体が常時適用の選好だけで構成されているか。肥大化していれば、観点2に基づき `docs/` への切り出しを提案。

## 提案のまとめ方

- 各観点の問題点と修正方針を整理してユーザーに提示する。
- 意図に関わる修正（禁止事項・自律性方針の変更等）は勝手に決めず確認する。
- 環境・ツールとの不整合の修正は、環境の読み取りで裏が取れているものを提案する。
