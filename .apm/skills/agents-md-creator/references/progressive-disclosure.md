# 段階的開示の切り出し基準

AGENTS.md本体と切り出し先 `docs` のどちらに情報を置くかを判定する基準。

切り出し先のディレクトリはモードで異なる。

- **リポジトリモード**: リポジトリルートの `.agents/docs/`。
- **グローバルモード**: 実体のAGENTS.mdと同じディレクトリ直下の `docs/`（例: `~/.config/agents/docs/`）。詳細は [global-entrypoint.md](global-entrypoint.md) 参照。

以下では切り出し先を `.agents/docs/` と表記するが、グローバルモードでは上記 `docs/` に読み替える。

## 判定軸は1つ:「常に読むべきか」

- **常時適用** → AGENTS.md本体に置く。
- **特定タスク時のみ必要** → `.agents/docs/` に切り出す。

この単一軸だけで振り分ける。複数の基準で迷わない。

## 本体に置くもの（常時適用）

どのタスクをするエージェントも知っておくべき情報。

- プロジェクト概要（全体像）。
- 守るべき原則・コーディング規約。
- 禁止事項・制約。
- 代表的なビルド・テスト・lintコマンド（日常的に使うもの）。
- 主要ディレクトリの地図。

## `.agents/docs/` に切り出すもの（特定タスク時のみ）

そのタスクをする時だけ参照すれば足りる情報。

- 詳細な手順（環境別ビルド手順、リリース手順、デプロイ手順等）。
- 具体例・サンプルコード。
- 状況依存ガイド（特定機能の実装ガイド、特定バグの対処法等）。
- 込み入った設定の説明。
- 滅多に使わないコマンドや運用。

## 切り出しの進め方

1. 各情報について「どのタスクでも常に読むべきか」を問う。
2. 常時適用なら本体に残す。特定タスク時のみなら `.agents/docs/<topic>.md` に移す。
3. 本体には、切り出した内容への**プレーン参照**だけを残す。これがデフォルト。
   プレーン参照の書き方は**Markdownリンク形式 `[リリース手順](.agents/docs/release.md)` を推奨**する（クリック可能でエディタ・IDEから辿れ、リンク切れも見つけやすい）。
   素のパス記述（例:「リリース手順は .agents/docs/release.md を参照」）も許容するが、推奨はMarkdownリンク。
   `@`インポート構文は自動展開されて段階的開示にならないため、必要時のみ読めばよい情報には使わない。
4. ただし、その内容を**どのタスクでも常時コンテキストに読み込ませたい**という設計意図がある場合は `@`参照を使ってよい。
   使い分けは「必要時のみ読めばよい → プレーン参照」「常に載せたい → `@`参照も可」。`@`参照の自動展開対応はツールにより異なり、Claude以外のエージェントでは無効になりうる点に留意する。

## ファイル分割の粒度

- `.agents/docs/` 配下はトピック単位で分ける（`build.md`、`release.md`、`testing.md` 等）。
- 1セクション内に常時適用と状況依存が混在する場合、常時適用分を本体に残し、状況依存分だけを切り出す。

## 本体の肥大化を避ける

- 本体が長くなったら、状況依存情報が混入していないか見直す。
- 設定ファイルで機械的に強制される規約は「設定に従う」と短く書き、詳細は設定ファイル自体に委ねる。
- 本体は常時適用のコアだけを簡潔に保ち、詳細は `docs/` へリンクで委譲する（必要時にのみロードされるオーケストレータ型）。指示を積み上げ続けると重要な指示が薄まり、トークンも浪費する。
- **削除基準**: 各行について「この行を消すとエージェントが具体的なミスを起こすか?」を問う。Yes なら残す、No/たぶん なら削除する。
- 古い/非推奨の記述は残さない。誤った指示は無いより害になる。
